# 0.1.25 migration notes

## What was broken

Built-in `post_round_hook` plugins (`claude_error_detector`,
`gemini_error_detector`) read `ctx.log_dir / f"round-{ctx.round_num}.log"`
to access the agent's JSONL output. That path is actually the supervisor's
wrapper subprocess stdout (always empty). The agent's real JSONL output
lives at `ctx.log_dir / "rounds" / f"R{round_num}-{timestamp}.log"`,
written by `runner.py`.

Result: since 0.1.20, all of these were silent no-ops in real
`agent-runner serve` use:

| Release | Feature | Real behavior pre-0.1.25 |
|---|---|---|
| 0.1.20 | Claude 5h rate-limit auto-detection + back-off | Plugin never fires |
| 0.1.23 | Transient error classification (4 buckets) | Plugin never fires |
| 0.1.24 (claude) | `agent_usage_recorded` event emission | Plugin never fires |
| 0.1.24 (gemini) | Same as above for gemini | Plugin never fires |

The bug was masked because all unit tests pre-seeded `round-{N}.log`
directly (the same wrong path the plugin assumed), so tests passed while
production produced no events. The 0.1.25 release adds an integration
test using real `agent-runner serve` to prevent this class of bug going
forward.

## The fix

New `HookContext.agent_log_path: Path | None = None` field, populated by
the supervisor at hook invocation. Built-in plugins now use it:

```python
def after_round(self, ctx: HookContext, result: Any) -> None:
    if ctx.agent_name != "claude":  # or "gemini"
        return
    log_path = ctx.agent_log_path
    if log_path is None or not log_path.exists():
        return
    # ... parse log_path ...
```

## Migration for 3rd-party plugin authors

If your plugin reads agent stdout from `ctx.log_dir / f"round-{N}.log"`,
your plugin has had this same silent-no-op bug since 0.1.20. To fix:

```python
# OLD (silently broken since 0.1.20)
log_path = ctx.log_dir / f"round-{ctx.round_num}.log"
if not log_path.exists():
    return

# NEW (0.1.25+)
log_path = ctx.agent_log_path
if log_path is None or not log_path.exists():
    return
```

If your unit tests pre-seeded the wrong path, update them to write to
`log_dir/rounds/R{round_num}-{anything}.log` instead. The in-tree
`tests/_test_helpers.py` ships an updated `write_round_log` helper if
you're testing inside the agent-runner repo.

If you've been wondering why your custom rate-limit detector never
seemed to fire — it's this bug. Apply the migration and your plugin
should start producing events on the next round.

## What did NOT change

- TOML config schema (no changes)
- Event payloads (no changes)
- Plugin entry-point names (no changes)
- `[plugins] disable = [...]` semantics (no changes)
- The `HookContext` fields you were already using (additive: new field
  has `= None` default; existing manual constructions still work)

## What's NOT in this release

- **Usage aggregation views** — deferred to 0.1.26
- **Tier-aware budget warnings** — 0.1.26
- **HTTP monitor usage panel** — 0.1.26
- **Dual-write to old `round-N.log` path for back-compat** — rejected:
  the old path had a different meaning (wrapper subprocess stdout); dual
  writing would conflict semantics in confusing ways
- **Auto-detect plugins using old path + warn** — rejected: too magic;
  explicit migration recipe is cleaner

## What downstream + other third-party users should do

1. Update to agent-runner 0.1.25
2. Audit any custom `post_round_hook` plugins for `ctx.log_dir /
   f"round-{N}.log"` and switch to `ctx.agent_log_path`
3. Re-run a `serve` round and check `events.jsonl` — features you
   thought were working (rate-limit detection, custom usage tracking)
   should now actually fire
