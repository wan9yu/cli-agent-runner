# 0.1.22 migration notes

## What changed

### 1. `[runtime] substrate_fingerprint_paths` — opt-in file-tree hash

```toml
[runtime]
substrate_fingerprint_paths = ["*.py", "specs/*.md"]
```

When configured, each round computes SHA-256 of the matched files'
contents (deduplicated, sorted) and emits it as `paths_hash` in the
`round_substrate_before` / `round_substrate_after` events.

When NOT configured (default `[]`): `paths_hash` is `null`. No file I/O cost.

### 2. `[runtime] fresh_eyes_every_n` — periodic fresh-eyes trigger

```toml
[runtime]
fresh_eyes_every_n = 50
```

Every 50th round (round_num multiple of N, and round_num > 0), the
supervisor sets `AGENT_RUNNER_FRESH_EYES=1` in the round subprocess env.
Otherwise it's `"0"` (always defined; matches `AGENT_RUNNER_PHASE` pattern).

When `fresh_eyes_every_n` is unset (default None): env var always `"0"`,
no `fresh_eyes_round_triggered` events emitted.

### 3. Three new events

| Event | Payload |
|---|---|
| `round_substrate_before` | `{round_num, git_head, paths_hash}` — emitted before round subprocess launch |
| `round_substrate_after` | `{round_num, git_head, paths_hash}` — emitted after round subprocess returns |
| `fresh_eyes_round_triggered` | `{round_num, every_n}` — emitted only on triggered rounds |

### 4. Always-on behavior change

`round_substrate_before` and `round_substrate_after` are emitted **every
round**, regardless of config. When neither `.git` exists nor
`substrate_fingerprint_paths` is set, both payload fields are `null`.
Negligible cost: only filesystem stat + 2 small JSONL appends per round.

If `.git` exists: `git_head` always captured via `git rev-parse HEAD`
(~5-10ms subprocess per round, twice). Negligible at round-cadence
timescales (rounds take minutes).

### 5. New `AGENT_RUNNER_FRESH_EYES` env var

Always set on each round subprocess. Values:
- `"0"` — normal round
- `"1"` — fresh-eyes round (triggered by `fresh_eyes_every_n` modulo)

Joins existing `AGENT_RUNNER_LOG_DIR` (0.1.15), `AGENT_RUNNER_ROUND_NUM`
(0.1.17), `AGENT_RUNNER_PHASE` (0.1.17).

### 6. New docs `docs/long-running-agents.md`

Covers the systemic hazards (confabulation, momentum drift, lineage rot),
the 4 framework primitives that provide raw data, the explicit abstraction
boundary (framework doesn't do semantic detection), and recommended
prompt-engineering practices.

**Read this before deploying agent-runner for any workflow expected to
exceed ~30 rounds.**

## Migration

None required. All additive. Opt-in via:
- `[runtime] substrate_fingerprint_paths = [...]` to enable paths_hash
- `[runtime] fresh_eyes_every_n = N` to enable fresh-eyes signal
- Plugins/prompts read the new events and env var as desired

## Opt-out

The always-on substrate events have no opt-out in 0.1.22 (events are
~negligible cost). If you need to suppress them entirely (extreme noise
sensitivity), file an issue; a `substrate_fingerprint_enabled: bool`
master switch could land in a future release.

## Background

This release is the agent-runner response to a 2026-05-16 hazard analysis
reporting momentum-driven confabulation in long agent-runner lineages
(stress test where the agent fabricated source-file LOC deltas and commit
SHAs without grounding). agent-runner ships the **infrastructure primitives**
(substrate fingerprint, fresh-eyes signal); the semantic detection layer
(attestation conventions, confabulation alarms) is deliberately left to
consumers' prompt-engineering layer. See `docs/long-running-agents.md`
§ "What agent-runner does NOT do" for the architectural rationale.
