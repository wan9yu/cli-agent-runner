# 0.1.21 migration notes

## What changed

### 1. `[runtime] max_rounds` + `--max-rounds N`

Supervisor exits cleanly after N round completions instead of running
infinitely. Counter is fresh per `agent-runner serve` invocation (does
not persist across supervisor restart).

```toml
[runtime]
max_rounds = 3      # exit after 3 completed rounds
```

Or via CLI (overrides config):

```bash
agent-runner serve --max-rounds 3 --config ./test.toml
```

Hit → emits `max_rounds_reached` event with payload
`{rounds_completed, max_rounds}` → `sys.exit(0)`.

**TOTAL rounds, not per-phase, not cycles.** To run N complete rotations
of a phase list: `max_rounds = N * len(phases.list)`.

### 2. `[runtime] stop_file` graceful pause sentinel

Operator-facing pause primitive. When set, supervisor checks the file's
existence between rounds. Found → finish current round → emit
`stop_file_detected` event → `sys.exit(0)`.

```toml
[runtime]
stop_file = "logs/stop-requested"   # relative paths resolve against work_dir
# stop_file = "/srv/agent/stop"     # absolute paths also supported
```

Ops workflow:

```bash
touch ~/.agent-runner/<project>/logs/stop-requested
# Current round finishes, supervisor exits cleanly (code 0)
sudo systemctl status agent-runner@<project>   # confirm clean exit
rm ~/.agent-runner/<project>/logs/stop-requested   # cleanup the sentinel

# To resume:
sudo systemctl start agent-runner@<project>    # explicit fresh start
```

**Important**: deleting the sentinel does NOT auto-resume. Requires
explicit `systemctl start` (or equivalent). This is intentional — ensures
operator pauses are deliberate state transitions.

### 3. New events

- `max_rounds_reached` — payload `{rounds_completed: int, max_rounds: int}`
- `stop_file_detected` — payload `{stop_file: str, content: str (≤200 chars), rounds_completed: int}`

Both indicate clean shutdown (exit 0). For `Restart=on-failure` systemd
units, these will NOT trigger respawn.

### 4. Three stop triggers, one clean-exit path

agent-runner 0.1.21+ has three between-rounds stop triggers that all
exit code 0 with distinct events:

| Trigger | Source | Event |
|---|---|---|
| `.agent-done` sentinel | Agent writes the file | `agent_self_terminated` (0.1.15) |
| `stop_file` sentinel | Operator touches the file | `stop_file_detected` (0.1.21) |
| `max_rounds` counter | Config / CLI | `max_rounds_reached` (0.1.21) |

Check order (first hit wins, single event emitted): `.agent-done` →
`stop_file` → `max_rounds`.

### 5. systemd unit pattern recommendations

For bounded runs (stress tests, batch jobs, migration validation),
switch `Restart=always` → `Restart=on-failure` so clean exits don't
respawn:

```ini
# Bounded job: 3 rounds and stop
[Service]
ExecStart=... serve --config /etc/agent-runner.toml --max-rounds 3
Restart=on-failure
RestartSec=5
```

Prod supervisor pattern (unchanged from current):

```ini
# Infinite supervisor
[Service]
ExecStart=... serve --config /etc/agent-runner.toml
Restart=always
RestartSec=5
```

## Migration

None required. All additive. Old TOMLs and old systemd units
(`Restart=always`, no `--max-rounds`, no `stop_file`) keep current
infinite-supervisor behavior.

## Opt-in

```toml
[runtime]
max_rounds = 3                       # bound this run
stop_file = "logs/stop-requested"    # enable operator pause
```

Or for one-off bounded runs (no TOML edit):

```bash
agent-runner serve --max-rounds 3 --config ./agent-runner.toml
```

## Relationship to `--once`

`--max-rounds 1` produces the same behavior as the existing `--once`
flag. `--once` remains supported (no deprecation); `--max-rounds N` is
the n≥2 generalization for "run a few rounds, exit."
