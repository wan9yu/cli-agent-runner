# Migrating to 0.1.32

## What changed

0.1.32 ships three Tier 2 additive features and one new monitor detector. All
new features default OFF or default-preserving — no config migration required.
Existing deployments are unaffected unless you explicitly opt in.

## Round-progress heartbeat: `round_progress_interval_s`

Mid-round visibility for long-running agents. When enabled, the supervisor
emits a `round_progress` event every N seconds while the agent subprocess is
live: `log_size_kb`, `last_write_age_s`, `wall_age_s`. Pairs with monitor
to distinguish "agent thinking" from "agent stuck with no log activity."

**Enable** (recommended for agents with rounds exceeding 10 minutes):

```toml
[monitor]
round_progress_interval_s = 300  # heartbeat every 5 minutes
```

For hour-long agents, 300–600 seconds is a reasonable interval. Shorter
intervals add minor I/O overhead (log stat call per tick).

**Event emitted**: `round_progress` (payload: `round_num`, `log_size_kb`,
`last_write_age_s`, `wall_age_s`).

**Default**: `0` (disabled). Set to the desired interval in seconds.

## `peek --select events.<kind>` query selector

Native events.jsonl query without shell pipelines:

```bash
# Last 10 agent usage records
agent-runner peek --select events.agent_usage_recorded

# Last 20 transient errors
agent-runner peek --select events.transient_error_detected --window 20

# Last 5 round progress heartbeats (when enabled)
agent-runner peek --select events.round_progress --window 5 --json | jq '.[] | .wall_age_s'
```

**Limitation**: reads the current month's events file only
(`events-YYYY-MM.jsonl`). Cross-month queries require direct file access.

**Output**: JSON array when `--json`; one JSON object per line otherwise.
Integrates cleanly with `jq` for field extraction.

## Host-health threshold config: `[monitor.host_health]`

Previously hardcoded thresholds for `mem_pressure`, `disk_warning`, and
`disk_critical` detectors are now configurable per deployment.

**Override example** (low-memory host):

```toml
[monitor.host_health]
mem_avail_min_mb = 500      # fire earlier than default 200
disk_warning_pct = 85.0     # warn at 85% instead of default 90%
disk_critical_pct = 92.0    # stop at 92% instead of default 95%
```

**Recommended values by environment**:

| Environment | `mem_avail_min_mb` | `disk_warning_pct` | `disk_critical_pct` |
|---|---|---|---|
| Raspberry Pi / small VPS | 300–500 | 80–85 | 90 |
| Cloud VM (4 GB+ RAM) | 200 (default) | 90 (default) | 95 (default) |

**Default**: all fields default to the previously hardcoded values — no
behavior change unless an operator adds a `[monitor.host_health]` section.

## `anomaly_repetitive_active` detector

Activates 0.1.31's `anomaly_repetitive_tool` event in monitor's alert flow.
Without this detector, the anomaly event was passive (visible in events.jsonl
but required manual grep). The new detector fires a `warning`-severity alert
when anomaly events appear in recent rounds.

**Behavior**: fires when `anomaly_repetitive_tool` events appear in the last
`window_rounds` rounds (default: 5). Notify-only — does not auto-stop.

**Tuning** (if default is too sensitive for your workload):

The detector uses hardcoded defaults in v1. If you observe false positives,
consider raising `anomaly_repetitive_threshold` (0.1.31 config) so fewer
events are emitted in the first place:

```toml
[monitor]
anomaly_repetitive_window = 10
anomaly_repetitive_threshold = 10  # require 10 repetitions, not default 8
```

Reducing `anomaly_repetitive_window` or raising `anomaly_repetitive_threshold`
reduces false positives at the event-emission level, which in turn quiets
the `anomaly_repetitive_active` detector.

**Detector count**: 10 → 11. README and `docs/commands.md` updated.
