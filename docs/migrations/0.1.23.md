## What changed

### 1. Unified transient-error classifier

agent-runner 0.1.20 shipped `rate_limit_rejected` / `rate_limit_recovered`
events + `rate_limit_action` config for the claude.ai 5h OAuth quota
specifically. 0.1.23 generalizes this machinery to cover all claude
transient API errors via a unified classifier:

| Classification | Trigger | Back-off |
|---|---|---|
| `rate_limit_account` | `rate_limit_event.status="rejected"` (5h quota) | exact `resetsAt` epoch |
| `rate_limit_model` | `api_error_status=429` without rate_limit_event | 60s |
| `api_transient_5xx` | `api_error_status` in {500, 502, 503, 504} | 60s |
| `api_timeout` | `api_error_status=408` | 30s |

Same supervisor mechanism (sleep until `reset_at_epoch + jitter`, 8h cap).
All 4 classifications honor `[runtime] transient_error_action`.

Motivation: Argus Gateway reported a 15-min claude API 500 outage that caused
16 wasted empty rounds before the operator noticed and intervened. The existing
rate-limit machinery already had the right back-off design; extending it to
cover 5xx/timeout was the natural generalization.

### 2. New events (additive)

```jsonc
// transient_error_detected
{"event": "transient_error_detected", "classification": "api_transient_5xx",
 "agent": "claude", "reset_at_epoch": 1778903400, "round_num": 42, "raw": "..."}

// transient_error_recovered
{"event": "transient_error_recovered", "classification": "api_transient_5xx",
 "agent": "claude", "throttled_for_s": 60}

// transient_error_backoff_capped (defensive — only fires if computed sleep > 8h)
{"event": "transient_error_backoff_capped", "classification": "rate_limit_account",
 "agent": "claude", "requested_sleep_s": 86400, "applied_sleep_s": 28800}
```

### 3. `runtime.rate_limit_action` deprecated -> `runtime.transient_error_action`

```toml
# Old (0.1.20)
[runtime]
rate_limit_action = "back_off"

# New (0.1.23) — both work in 0.1.23, only new works in 0.1.24+
[runtime]
transient_error_action = "back_off"
```

Setting `rate_limit_action` triggers a DeprecationWarning. Setting BOTH
raises `ValueError`. Same 3 values (`back_off` / `skip` / `stop`),
same semantics — only difference is `transient_error_action` applies to
all 4 classifications, not just 5h rate limits.

### 4. 0.1.20 events kept as aliases (1-release deprecation window)

For `rate_limit_account` classification ONLY, agent-runner 0.1.23 emits
BOTH the new `transient_error_detected` event AND the old
`rate_limit_rejected` event. Same for `_recovered` and `_backoff_capped`.

Operators with plugins subscribed to old event names (`rate_limit_rejected`):
- Keep working in 0.1.23 for 5h rate-limit case
- **Miss** the new 5xx / 429-model / 408-timeout events
- Should migrate to subscribe to `transient_error_detected` and dispatch on
  `classification` field

0.1.24 drops the dual-emit.

### 5. Plugin rename (entry-point alias preserved)

`claude_rate_limit_detector` is renamed to `claude_error_detector`. The
0.1.20 entry-point name is kept as an alias so:
- `[plugins] disable = ["claude_rate_limit_detector"]` continues to work
- `[plugins] disable = ["claude_error_detector"]` also works (canonical)

In `pyproject.toml` both entry points point to the same class; the
hook's `name` attribute is `claude_error_detector` only.

## Migration

For operators with `rate_limit_action` in TOML: change to `transient_error_action`
(same value works). DeprecationWarning until you migrate.

For plugin authors subscribed to old events: add a handler for the new
`transient_error_detected` event that dispatches on `classification`.
The old events keep firing for `rate_limit_account` through 0.1.23.

For `[plugins] disable = ["claude_rate_limit_detector"]`: no action needed
(alias resolves to new hook).

## What did NOT change

- Supervisor back-off mechanism (sleep until `reset_at_epoch + jitter`)
- Default `back_off` action
- 8h sleep cap
- `peek --json` schema (still 1.9; `rate_limit` field semantically widened to
  reflect any classification — may be revisited in 0.1.24)
- Claude OAuth account quota handling (same precision via `resetsAt`)

## What is NOT in this release

- `auth_fail` structured detection from `api_error_status=401`: kept on
  existing `monitor.auth_fail_patterns` regex path. Adding it to transient_error
  family would muddy the "transient = recoverable via sleep" semantic.
- Per-class action overrides (`api_transient_5xx = "stop"`, etc.): single
  knob applies uniformly across all classifications. Add later if demand surfaces.
- Exponential back-off with consecutive-error counting: flat defaults (60s/30s)
  are simpler and bounded. Revisit if real data shows flat is wrong shape.
- aider / gemini error classifiers: separate plugins (future releases). The
  generic `transient_error_*` event family already supports them without core
  changes.
