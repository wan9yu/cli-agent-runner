# Migration guide: 0.2.2

## TL;DR

Two public symbols are gone: `agent_runner.detector_helpers` and `cancel`
(both the CLI verb and `api.cancel()`). If you import neither, no code change
is needed and no TOML change is required.

**If any plugin calls `register_plugin_owned_paths`, read the next section
before upgrading.** You may have deliverables sitting in old orphan stashes
right now.

Config validation also got stricter: a config that was silently broken before
now fails loudly at load. That is the fix, not a regression — see
[Stricter config validation](#stricter-config-validation).

---

## Fixed: plugin-owned paths were swept into the orphan stash

**This is the most important entry in this guide.**

`register_plugin_owned_paths` was honored at the report boundary and ignored at
the git boundary. `detect_dirty_files` filtered your owned paths out of the
dirty-file report, but `stash_orphan`'s pathspec never read the registry — so
`git stash push -u` took the owned files anyway.

The guard held only when owned paths were the *only* dirty files. The moment the
agent also touched one non-owned file — the common case — the deliverable went
into the stash and off disk.

**It was silent by construction.** `OrphanState.files` and the `orphan_stashed`
event are built from the *filtered* list, so `events-*.jsonl`, `orphan-state.json`
and `peek` all omitted the swept file. The stash contained a file nothing told
you about.

0.2.2 makes the registry authoritative at the git boundary: `stash_orphan`
extends its pathspec with an `:(exclude)` entry per registered pattern, built by
running the concrete dirty paths back through the same `_matches_owned_path`
matcher the report boundary uses, and gated on `git check-ignore` exactly as the
`log_dir` exclusion already was. All three documented pattern forms are now
honored on both boundaries:

| Pattern form | Example | Matches |
|---|---|---|
| Trailing-slash prefix | `proposals/` | `proposals/` and anything under it |
| Glob (`PurePath.match`) | `reports/*.md` | `*.md` directly in `reports/`; `*` does not cross `/` |
| Globstar (`**`) | `logs/plugins/**/*` | files directly in `logs/plugins` **and** nested below it |

### Recovering files swept before 0.2.2

The stashes are still there — nothing deleted them.

```bash
# 1. List them. agent-runner's stashes carry an ORPHAN R<N> <phase> subject.
git stash list

# 2. Inspect one. --include-untracked is required: a deliverable that was
#    untracked when it was swept does not appear without it.
git stash show --include-untracked --name-only 'stash@{0}'

# 3a. Restore a file that was TRACKED and modified when it was stashed:
git checkout 'stash@{0}' -- proposals/memo.md

# 3b. Restore a file that was UNTRACKED when it was stashed. Step 3a fails on
#     these with "pathspec ... did not match any file(s) known to git" —
#     untracked entries live in the stash's third parent:
git checkout 'stash@{0}^3' -- proposals/memo.md
```

Work backwards from the newest stash. Restore individual paths rather than
`git stash pop` — a pop also restores the round's agent WIP into your tree.

## Changed: `**/*` owned-path patterns match direct children

Independently of the sweep fix above, the `**` matcher itself changed, and this
changes which paths get filtered for **existing** plugin users.

Before 0.2.2, `**` patterns were matched with `fnmatch`, whose `**/` compiled to
a regex that demanded an intervening directory segment. So
`logs/plugins/my_plugin/**/*` matched `logs/plugins/my_plugin/sub/state.json`
but **not** `logs/plugins/my_plugin/state.json` — a file sitting directly in the
registered directory was not owned, and was swept.

0.2.2 matches `**/` as **zero or more** directory segments (git/bash `globstar`
semantics), so `<dir>/**/*` now matches both a direct child and a nested file.
`reports/**/*.md` now matches both `reports/dev.md` and `reports/sub/qa.md`.

If you registered a `**/*` pattern expecting the old behavior, `detect_dirty_files`
now filters more paths than before — the paths it was previously letting through
were the ones the sweep bug then lost, so this is the intended fix, but the change
in the filtered set is worth knowing.

(Globstar `?` matches one non-slash character; single `*` still does not cross a
slash; character classes inside `**` patterns are not supported.)

## Removed: `agent_runner.detector_helpers`

**Breaking.** The module and its three public helpers are gone:

| Removed | Was |
|---|---|
| `cumulative_window_check` | shared detector windowing helper |
| `dual_source_silence` | shared two-source silence check |
| `phase_filter` | shared phase-scoped event filter |

Nothing in-tree imported them; no built-in detector used them. They were
published in 0.1.24 as a plugin-author convenience. If an out-of-tree detector
does import them, vendor the ~115 LOC into your own package — the `Detector`
Protocol (`name` / `severity` / `auto_action` + `detect(self, state)`) is
unchanged, so no other edit is needed.

## Removed: `cancel`

**Breaking.** Both surfaces are gone:

- the `agent-runner cancel` CLI verb
- `agent_runner.api.cancel(project)`

**The honest reason**: it never did what it documented. `cancel` was specified as
"SIGINT to the current round, a commit-and-exit hint". It read a `round.pid` file
that nothing in the package ever wrote, so the SIGINT forward was unreachable on
every invocation. Its only real effect was setting the graceful-stop flag —
making it a slower alias for `stop`.

**Use `stop`.** It is what `cancel` actually did, and what it does is documented:

```bash
agent-runner stop      # graceful: serve exits after the current round completes
agent-runner kill      # forced: SIGTERM, then SIGKILL after 5s
```

The `SIGUSR1` handler on `serve` is removed with it. `serve` still writes
`serve.pid`; it no longer writes (or claims to write) `round.pid`.

## Fixed: `orphan_stash_failed` and `orphan_idempotent_skip` now actually fire

Both kinds have been declared and published as built-in event kinds for
releases, and neither was ever emitted. **Consumers may now see event kinds they
have never received before.**

| Event | Now emitted when |
|---|---|
| `orphan_stash_failed` | `git stash push` exits non-zero. Previously `stash_orphan` returned `None` here, which the default dirty handler mapped to `DirtyOutcome(kind="ignored")` — indistinguishable from an operator choosing `dirty_action = "ignore"`, while the WIP carried silently into the next round. `stash_orphan` now raises `StashError` and the handler emits this event. |
| `orphan_idempotent_skip` | A second `stash_orphan` in the same idempotency window reuses the existing stash (`StashRef.reused` is `True`). Previously this re-emitted a duplicate `orphan_stashed`. |

The push failure is not theoretical: `git add -N` (intent-to-add) followed by
`git stash push -u` returns rc=1 with `error: Entry '<file>' not uptodate.
Cannot merge.`

### `StashError` import note

`api.stash_orphan` re-exports the function, but **not** `StashError`. Import the
exception from where it is defined:

```python
from agent_runner.api import stash_orphan
from agent_runner.vcs_state import StashError   # not re-exported from api
```

This matches the existing `try_auto_commit` / `AutoCommitError` precedent — the
primitive is a public `api` symbol, its exception type lives in `vcs_state`.

If you subscribe to `events-*.jsonl` and switch on `kind`, make sure an unknown
kind is ignored rather than fatal — that has always been the contract, but these
two are the first to exercise it in a while.

## Stricter config validation

Three fields bypassed validation and are now checked at load. **A config that
was silently broken now fails loudly at startup.** That is the fix, not a
regression — each of these was already wrong, just invisibly.

| Field | Was | Now |
|---|---|---|
| `[monitor.host_health] disk_warning_pct` / `disk_critical_pct` | bare `float()`: `true` → `1.0`, `500.0` accepted | must be a number in `0.0 .. 100.0`; bool rejected |
| `[prompt] inject_context` | bare `bool()`: `"false"` → `True` | must be a TOML boolean |
| `[monitor] auth_fail_patterns` | not compiled until first use; a bad regex crashed the monitor mid-run | compiled at load, matching `[runtime] grace_kill_ignore_patterns` |

The `disk_critical_pct` case had teeth: `disk_critical` carries
`auto_action = "stop_service"`. A threshold of `500.0` was accepted and the
detector then never alerted, at any disk usage. If your config fails to load
after upgrading, the failure is telling you a defense was off.

## Changed: `ConfigError` is raised consistently

Config-load rejections now raise `ConfigError` at the user-facing sites that
previously raised a bare `ValueError`. This is a **widening, not a break**:
`ConfigError` subclasses `ValueError`, so any existing `except ValueError`
around `load_config` still catches it. Code that already catches `ConfigError`
now catches more of what the docstring always promised.

---

## Removed: `events.SIGTERM_RECEIVED`

The constant (value `"sigterm_received"`) was a public module name, but it was
never emitted by any code path in any release — its premise was a bash-wrapper
respawn loop retired long ago. No consumer can have received it. `serve`'s
graceful-stop handler is and was the real mechanism.

If you registered a subscriber for `sigterm_received`, it has never fired;
delete it. Subscribe to `serve_stopping` / `round_end` instead.

The Chinese README was a fork of the English one and had drifted: it still
stated a "12 defenses" count against the English "11 detectors", and it linked

## Out-of-tree consumers we cannot see

This repository cannot tell whether anyone outside it imports `detector_helpers`,
calls `api.cancel()`, subscribes to `sigterm_received`, or reads
`cfg.plugins.raw`. The removals above were made on in-repo evidence — zero
in-tree callers, and in `cancel`'s and `sigterm_received`'s case, a mechanism
that provably never ran. `cfg.plugins.raw` is the inverse: it has no in-tree
reader either, but it is **kept** on purpose as a plugin-author passthrough
contract, precisely because a plugin package may read its own `[plugins.*]`
sub-keys through it.

If one of these removals breaks you — or if you rely on `cfg.plugins.raw` and
want it to stay — please
[open an issue](https://github.com/wan9yu/cli-agent-runner/issues) with your use
case. A removal is reversible; a removal made without hearing from you is the
part we would like to fix.
