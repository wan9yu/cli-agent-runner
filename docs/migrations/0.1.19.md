# 0.1.19 migration notes

## What changed in `agent-runner install`

### 1. ExecStart now uses actual script path

Before 0.1.19, `agent-runner install` wrote
`ExecStart=/usr/bin/agent-runner serve ...` when invoked under
`pip install --user` (because the implementation derived the script
location from `Path(sys.executable).parent`, which is the Python
interpreter's dir, not the script's dir). On Debian / dietpi / RPi OS,
the script lives at `~/.local/bin/agent-runner`, so the resulting unit
failed to start with `Status=203/EXEC`.

0.1.19 uses `shutil.which("agent-runner")` (with `sysconfig` fallback)
to locate the actual script. If neither resolves, install fails fast
with `FileNotFoundError`.

### 2. Fail-fast when user systemd is unavailable

Before 0.1.19, `agent-runner install` ran `systemctl --user enable` /
`start` with `check=False`. On distros without a user D-Bus session
(headless dietpi, RPi OS Lite without lingering), all four systemctl
calls silently failed but `install()` still returned
`enabled=True, started=True`. Operators thought install succeeded.

0.1.19 pre-flights `XDG_RUNTIME_DIR` and `systemctl --user is-system-running`
before writing unit files. If user systemd is unavailable, install exits
1 with a hint:

> user systemd unavailable (D-Bus session not running). On headless
> distros, run `sudo loginctl enable-linger $USER` and re-login, OR pass
> `--system` for a system-level unit.

Operators who scripted around the old behavior (e.g. by assuming
non-zero exit means failure) — no change needed; this just makes the
behavior match expectation. Operators who relied on `install()` always
returning success — script needs to handle non-zero exit.

### 3. `--system` install mode shipped

`agent-runner install --system [--monitor]` writes
`/etc/systemd/system/agent-runner@<project>.service` with `User=$SUDO_USER`
+ `Group=$SUDO_USER` + `WantedBy=multi-user.target`. Enables but does
NOT auto-start (system-level deserves explicit operator decision).

Usage:

```bash
sudo -E agent-runner install --system --monitor
# Then start when ready:
sudo systemctl start agent-runner@<project>
```

`-E` preserves `SUDO_USER` env var. The process runs as your user (not
root) thanks to the `User=` directive.

When to use `--system`: dietpi, RPi OS Lite, fleet deployment, any
headless distro where user systemd is impractical.

See also `docs/runbook.md §Install prerequisites by distro`.
