# 0.1.20 migration notes

## What changed

### 1. Built-in claude rate-limit detection

When claude CLI returns a 429 rate-limit rejection in its JSONL output
stream, agent-runner 0.1.20+ detects it and automatically pauses round
dispatch until claude's `resetsAt` epoch. Pre-0.1.20 supervisors would
rapid-fire failed rounds through the throttle window producing zero work.

Three new events surface this:

- `rate_limit_rejected` — emitted by `claude_rate_limit_detector`
  post_round_hook when 429 detected in round log. Payload: `agent`,
  `reset_at_epoch`, `limit_type`, `round_num`, `raw` (≤200 chars).
- `rate_limit_recovered` — emitted by supervisor right before resuming
  after sleep. Payload: `agent`, `throttled_for_s`, `limit_type`.
- `rate_limit_backoff_capped` — defensive event when computed sleep
  exceeds 8h cap (malformed epoch or time drift). Payload: `agent`,
  `requested_sleep_s`, `applied_sleep_s`.

### 2. New config: `[runtime] rate_limit_action`

```toml
[runtime]
rate_limit_action = "back_off"   # default; see below for alternatives
```

| Value | Behavior |
|---|---|
| `back_off` (default) | Sleep until `reset_at_epoch + jitter (5–30s)`, then emit `rate_limit_recovered` and resume |
| `skip` | Emit event, immediately try next round (pre-0.1.20 behavior; useful for stress-tests) |
| `stop` | Emit event + `agent_self_terminated` with reason `"rate_limit"`, exit supervisor cleanly (batch-mode) |

### 3. `peek --json` schema 1.8 → 1.9 (additive)

New optional top-level field `rate_limit`:

```jsonc
{
  // ... existing 1.8 fields ...
  "schema_version": "1.9",
  "rate_limit": null              // when not throttled
  // or when throttled:
  "rate_limit": {
    "throttled_until_epoch": 1778903400,
    "limit_type": "five_hour",
    "agent": "claude",
    "since_round": 42
  }
}
```

Consumers that check `schema_version == "1.8"` strictly will need to
accept `"1.9"`. Consumers that check `>= "1.8"` require no change.

### 4. New monitor detector: `rate_limit_active`

A tenth built-in detector fires a `warning`-severity alert (no auto-stop)
while the supervisor is in the throttle window. The alert clears
automatically once `rate_limit_recovered` is emitted. No configuration
needed; `detect_rate_limit_active` runs on every monitor poll.

## Action required

No migration action is required for existing deployments. The three new
event kinds are appended to `events.jsonl` going forward; old logs are
not modified. The `rate_limit` field in `peek --json` is `null` when not
throttled, so existing consumers receive a compatible (additive) response.
