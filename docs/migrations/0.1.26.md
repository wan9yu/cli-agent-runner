# 0.1.26 migration notes

## What was broken

After the 0.1.25 plugin path hotfix restored the event-emission flow,
ship-day smoke tests on real claude-opus-4-7 and gemini v0.42.0
revealed three independent data-shape bugs in `agent_usage_recorded`
events. The 0.1.25 fix made events fire — now we fix what's in them.

### 1. Claude `model: "unknown"` (since 0.1.24)

The `claude_error_detector` plugin read
`result_event.get("message", {}).get("model")`, but claude's terminal
`result` event has no `message` field. The model lives in earlier
`assistant`-type events: `{"type":"assistant","message":{"model":"claude-opus-4-7",...}}`.

Plugin now scans the 50-line tail and tracks the latest `assistant.message.model`,
passing it as a separate kwarg to `_extract_usage`. Edge case (immediate
error before any assistant event) emits `"unknown"` as before.

### 2. Claude `input_tokens: 0` (since 0.1.24)

The 0.1.24 simplify pass was misled by a reviewer's claim that
Anthropic's `usage.input_tokens` is gross (includes cached tokens).
**That claim was wrong.** Per actual Anthropic API behavior and
documentation, `usage.input_tokens` is the NET fresh input count —
independent of `cache_read_input_tokens` and `cache_creation_input_tokens`.

The broken formula:

```python
input_net = max(usage["input_tokens"] - usage["cache_read_input_tokens"], 0)
```

produced 0 for every round with any cache hit. Real claude rounds with
cached prompts (the common case) reported `input_tokens=0` regardless
of actual usage.

The fix is to use the field directly:

```python
"input_tokens": int(usage.get("input_tokens", 0)),
```

**Migration impact for aggregation scripts**: any aggregation script
written against 0.1.24 data that sums `input_tokens` will see real
positive values starting in 0.1.26. The data was wrong before; this
is the correction, not a spike. Update baselines accordingly.

### 3. Gemini preset `-o text` (since 0.1.19)

The gemini preset shipped:

```toml
command = ["gemini", "--yolo", "--skip-trust", "-o", "text"]
```

`-o text` produces human-readable output. The `gemini_error_detector`
plugin (added 0.1.24) parses JSONL — incompatible with text format.
Result: `agent_usage_recorded` events never fired for gemini rounds.

Fixed by changing the preset:

```diff
-command = ["gemini", "--yolo", "--skip-trust", "-o", "text"]
+command = ["gemini", "--yolo", "--skip-trust", "-o", "stream-json"]
```

**Operator-visible change**: round log files
`log_dir/rounds/R<N>-<timestamp>.log` for gemini rounds now contain
JSONL instead of human-readable text. Operators who `tail -F` the raw
round log will see structured output (one JSON event per line).

For human-readable status, use `agent-runner monitor --mode http`
which renders the events into a browser-friendly page.

**Existing scaffolds need manual TOML edit** — only new scaffolds from
`agent-runner init --preset gemini` (0.1.26+) get the fix automatically.

## Migration

For claude operators: no action needed. Plugin fix is automatic on upgrade.

For gemini operators: edit `agent-runner.toml` and change `-o text` to
`-o stream-json` in `[agent].command`. Required for `agent_usage_recorded`
events to fire (they never did pre-0.1.26 for gemini).

For anyone with 0.1.24+ aggregation scripts: re-verify your `input_tokens`
field expectations. Claude values will jump from 0 to real positive
numbers; gemini events will start appearing. Aggregations should now
reflect actual consumption.

## Verifying the fix

After upgrading, run a single round and inspect the event:

```bash
agent-runner serve --max-rounds 1 --config agent-runner.toml
# After it exits:
grep "agent_usage_recorded" ~/.agent-runner/<project>/logs/events-*.jsonl | tail -1 | python3 -m json.tool
```

You should see:
- `"agent": "claude"` or `"agent": "gemini"`
- `"model"`: real model name (e.g. `"claude-opus-4-7"`, not `"unknown"`)
- `"input_tokens"`: real positive int
- `"cached_tokens"`: real int (may be 0 for first-ever round)
- `"cost_usd"`: real USD for claude, `null` for gemini

If any field is still wrong, file an issue with the round log file
contents (`~/.agent-runner/<project>/logs/rounds/R<N>-*.log`).

## What did NOT change

- `agent_usage_recorded` event schema (field names, field types)
- `HookContext`, all dataclasses
- TOML config schema
- Plugin entry-point names
- `[plugins] disable = [...]` semantics
- Architecture invariants (CLI-agnostic core, plugin-side CLI parsing)

## What's NOT in this release

- **Aggregation features** (`peek --json .usage`, tier-aware budget warnings,
  HTTP monitor usage panel, `agent-runner usage --since 24h` exporter) —
  deferred to 0.1.27 now that data is finally correct
- **`cache_creation_input_tokens` field** — add to schema when 0.1.27
  aggregation surfaces a need
- **Aider plugin** — same plugin family will work when a tester is available
